Correction Component

Text correction activity where students identify errors by striking through incorrect text and typing the correct replacement. Supports 1-10 correction sentences with configurable attempts and feedback.

OverviewUML ClassSequenceComponent FlowFeaturesTechnicalData ModelFilesSettingsBehaviors

Overview

The Correction Component presents sentences containing deliberate errors. Students must strikethrough the incorrect word or phrase and type the exact correct replacement. Authors configure 1-10 sentences, each with a list of errors and their correct answers. Validation is case-insensitive and whitespace-normalized.

Key Capabilities

  • 1-10 correction sentences per component
  • Students strikethrough errors and type replacements
  • Case-insensitive, whitespace-normalized comparison
  • 1-5 configurable attempt limit
  • Show Me reveals correct answers
  • SCORM state saved as raw HTML strings

Behaviors and Constraints

  • Dual action required: Students must BOTH strikethrough the incorrect text AND type the exact replacement. Either action alone is insufficient for scoring.
  • allowRestart=false forces maxTries=1: When Allow Activity Restart is unchecked, the maxTries is automatically set to 1, giving students only one submission.
  • SCORM XSS risk: State is saved as raw HTML strings via stateMainatainCorrection(). This means student input is stored without sanitization, posing a potential XSS vulnerability.
  • Global selectionchange listener: document.addEventListener("selectionchange") is registered globally and never removed, causing a performance issue when multiple correction components exist on a page.
  • Browser-specific span handling: IE detection code exists for creating strikethrough spans due to differences in how browsers handle execCommand("strikethrough").
  • Reset behavior: Reset keeps correctly identified errors visible (strikes remain) but clears incorrect strikes and replacement inputs.
